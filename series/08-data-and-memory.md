# 08｜Data & Memory：上下文不可靠，比没上下文更危险

企业 Agent 的失败往往不是“不会推理”，而是“上下文不完整、不一致或过期”。

在执行型系统里，**过期上下文**甚至比缺少上下文更危险，因为它可能让系统“自信地做错事”。

---

## 1) 三类上下文：事实 / 规则 / 记忆

- **Structured Data（事实）**：交易与主数据，一致性优先
- **Knowledge（规则/制度）**：SOP、规程、知识手册，可解释优先
- **Memory（连续性）**：任务历史与执行经验，复用与个性化优先

三者不能互相替代。

---

## 2) 检索链路：从“召回文本”升级为“可执行上下文包”

一个实用的链路通常包含：

1. 语义召回（找相关）
2. 结构化查询（补实时状态）
3. 约束注入（权限/规则/风险边界）
4. 上下文压缩（注入到计划与执行）

---

## 3) 新鲜度与一致性：必须显式化

你需要明确：

- 哪些字段必须实时回源
- 哪些允许缓存多久
- 哪些动作不能在最终一致语义下执行

否则 Agent 的执行会建立在模糊假设上。

---

## 4) 记忆治理：遗忘、纠偏与合规

Memory 不是“越多越好”，而是：

- 有有效期
- 可纠偏
- 可清理
- 可追踪访问
