# 05｜Interface 层：为什么 REST 还不够，需要 Discovery + Schema

> 适合发布：平台团队/集成团队

很多团队说“我们都有 REST API 了”，但 Agent 接入后仍然会频繁失败：

- 找不到能力（不可发现）
- 参数填错（语义不清）
- 错误处理混乱（错误语义不一致）
- 变更悄悄破坏线上任务（缺少版本/兼容策略）

对人类开发者来说，这些问题可以靠经验和沟通修补；对 Agent 来说，它们会直接变成成功率和稳定性问题。

---

## 1) REST 解决“可调用”，但没解决“可被 Agent 稳定调用”

Agent 需要的是：

- 能力在哪里（Discovery）
- 字段怎么填/怎么验（Schema）
- 失败怎么分类与恢复（标准错误语义）
- 权限怎么握手（统一鉴权）
- 接口如何演进（版本与兼容策略）

---

## 2) 一个现实组合：REST + Discovery + Schema

- REST：兼容性强、生态成熟
- Discovery：能力清单可获取、语义可解释、变更可感知
- Schema：输入输出机器可校验、减少隐式约定

---

## 3) 自查清单（可以直接贴到评审里）

- [ ] 关键能力是否可被统一检索/发现？
- [ ] 是否有机器可校验的 schema（输入/输出/枚举/错误）？
- [ ] 是否有统一鉴权与细粒度授权策略？
- [ ] 是否有幂等/重试/超时标准？
- [ ] 变更是否可观测，能否提前通知到依赖方（含 Agent）？
