# 06｜Capability 层：能力节点/能力注册/能力图，怎么做才不沦为文档工程

> 适合发布：业务架构/平台架构（高收藏）

Interface 层把系统“连起来”之后，很多团队会发现：依然很难让 Agent 稳定完成业务。

原因是：Agent 面对的仍然是一堆离散 API，而不是可规划的业务能力。

Capability 层的目标，是把“接口可调用”升级为“业务可完成”。

---

## 1) Capability 层不等于“再包一层 API 文档”

真正的 Capability Registry 更像控制平面：

- 生命周期管理：发布/灰度/下线
- 版本治理：兼容性、破坏性变更检测
- 策略绑定：权限/风控/审计随能力定义沉淀
- 发现与路由：为 Agent 提供可筛选的能力视图

---

## 2) 能力节点最小结构

建议每个能力节点至少写清：

- 目标语义
- 输入/输出 schema
- 前置条件
- 执行约束（幂等、超时、重试、补偿）
- 失败语义（可恢复/不可恢复）
- 治理信息（审计字段、风险等级、责任主体）

---

## 3) Capability Graph：把“流程经验”变成可计算资产

能力图至少要表达三类关系：

- 依赖（必须先执行）
- 替代（失败可切换）
- 约束（满足条件才能走该路径）

有了图，Agent 的规划不再是“现场猜”，而是“在约束内选路径”。

---

## 4) 建设路径（建议）

不要先做全企业大图，建议：

1. 选 1 条高价值链路
2. 抽 10~20 个高复用能力节点
3. 先把 schema + 策略 + 审计绑到能力层
4. 再逐步沉淀能力图与模板
