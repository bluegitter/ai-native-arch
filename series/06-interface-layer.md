# 06｜Interface 层：让 Agent 真的“连得上、调得动、调得稳”

Interface Layer 经常被低估。

很多团队觉得“只要有 API 就够了”，但 Agent 的大量失败其实发生在接口层：

- 找不到能力（不可发现）
- 参数构造错误（语义不清）
- 错误语义不一致（无法恢复）
- 权限握手复杂（无法自动化）

---

## 1) 面向 Agent 的接口，核心不是“转发”，而是“语义统一”

Interface 层建议至少覆盖：

- **Discovery**：能力清单、版本、下线状态
- **Schema**：输入输出可校验
- **AuthN/AuthZ**：与企业身份体系打通
- **Error model**：错误语义标准化（可重试/需补偿/需转人工）
- **Observability hooks**：调用链可关联到任务与主体

---

## 2) 为什么不推荐 CLI 作为主接口

CLI 对工程师高效，但对 Agent 不友好：

- 输出非稳定文本，解析脆弱
- 参数语义隐式依赖经验
- 权限/审计链路分散

CLI 适合作为兜底与人工运维，但不宜作为生产级 Agent 的主执行接口。

---

## 3) 一个现实组合：REST + Discovery + Schema

- REST：兼容性与生态
- Discovery：让 Agent 知道“有什么、在哪、变没变”
- Schema：让 Agent 知道“怎么填、怎么判、错了怎么办”
