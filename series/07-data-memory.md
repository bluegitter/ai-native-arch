# 07｜Data & Memory：上下文不可靠比模型不聪明更危险

> 适合发布：数据平台/知识库/Agent 平台团队

企业 Agent 的失败，常常不是“不会推理”，而是“上下文不完整、不一致或已过期”。

对企业来说，过期上下文比缺少上下文更危险：因为 Agent 可能据此触发真实动作。

---

## 1) 三类上下文：Structured Data / Knowledge / Memory

- **Structured Data**：交易与主数据（事实基础，强调一致性）
- **Knowledge**：制度、SOP、手册（规则与解释，强调可检索）
- **Memory**：任务历史与执行经验（连续性，强调可复用）

三者不能互相替代。

---

## 2) 检索链路：从“召回内容”到“可执行上下文包”

建议链路：

1. 语义召回（相关知识/历史）
2. 结构化查询（补齐实时状态）
3. 约束注入（权限、规则、风险边界）
4. 上下文压缩（注入计划与执行）

关键原则：**先约束后生成**。

---

## 3) 数据新鲜度与一致性：必须显式化

- 哪些字段必须实时？哪些可近实时？哪些可缓存？
- 哪些动作不能在最终一致语义下执行？

把这些要求作为 Agent 执行前置约束，而不是“数据平台内部指标”。

---

## 4) 记忆治理：遗忘、纠偏、合规

Memory 带来连续性，也带来风险：错误记忆/过期记忆/越权记忆都会放大偏差。
