# 02｜别再把 Agent / LLM / Workflow / RPA 混在一起：四个概念边界一张表

> 适合发在知乎回答：容易引发讨论与收藏。

企业讨论“上 Agent”时最常见的跑偏，不是技术路线，而是**概念混用**：

- 有人说的 Agent，其实是 LLM 问答
- 有人说的 Agent，其实是 Workflow 编排
- 有人说的 Agent，其实是 RPA 脚本

概念不清，会导致：目标错位、选型错位、治理缺位。

---

## 一张表：四者各自解决什么问题

| 概念 | 解决问题 | 核心优势 | 典型短板 |
|---|---|---|---|
| **LLM** | 理解与生成 | 语言理解、推理、总结、生成内容 | 不负责执行闭环与治理 |
| **Agent** | 目标驱动的持续执行 | 动态规划+工具调用+长任务推进 | 需要接口/治理/可观测底座 |
| **Workflow** | 固定流程编排 | 确定性、可追踪、强合规 | 对变化与异常分支适应差 |
| **RPA** | 操作自动化复刻 | 不改造系统也能“先跑起来” | 维护成本高、脆、泛化差 |

---

## 一个更现实的结论：不是四选一，而是分层协同

在多数企业里，更可行的形态往往是：

- **LLM** 提供认知与生成
- **Workflow** 承载稳定骨架（合规、审批、关键路径）
- **RPA** 覆盖遗留系统的过渡缺口
- **Agent** 负责跨系统、非结构化、异常分支与动态规划

你要避免的，是把其中任意一个当成另三个的“万能替代”。
