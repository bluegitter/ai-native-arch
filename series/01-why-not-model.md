# 01｜为什么 Agent 落地难点不在模型：企业系统要补的 4 个底座

> 适合发布：知乎回答 / 专栏短文 / 内部分享

很多团队聊 Agent 落地，第一反应是“换更强模型”“调提示词”“上多智能”。但在企业场景里，真正让 Agent 走不进生产环境的，往往不是模型能力，而是**系统不支持机器稳定执行**。

更直白一点：模型解决的是“会不会想”，企业要解决的是“能不能做、能不能持续做、出了事能不能解释”。

这也是为什么你会看到一种常见现象：PoC 演示很好看，一旦进入跨系统、跨部门、强合规的真实链路，系统立刻变脆。

下面用 4 个“底座”把问题说清楚（也是长文的主线）。

---

## 1. 底座一：API-first（关键业务动作要可调用）

如果关键动作仍然锁在页面按钮、人工表单、线下电话、临时脚本里——Agent 再聪明也只能停留在“建议”。

**判断标准（可自查）：**

- 关键业务动作是否有稳定 API（不是临时脚本/抓包接口）？
- 是否有清晰的权限边界和幂等/重试语义？
- 是否能返回“业务级结果”（而不是仅 200 OK）？

---

## 2. 底座二：Schema-first（业务语义要机器可校验）

传统接口文档很多是给人看的，靠经验补齐“字段含义/枚举范围/错误语义”。Agent 做不到靠经验补齐。

没有 schema，风险常常不是“调用失败”，而是更危险的：**调用成功但业务语义错了**。

**判断标准（可自查）：**

- 输入/输出是否有结构化契约（JSON Schema / OpenAPI / protobuf）？
- 枚举值、约束、错误码语义是否统一？
- 变更是否有兼容策略，能否提前预警？

---

## 3. 底座三：Capability-based（从接口清单升级为能力目录/能力图）

“接口很多”不等于“业务能完成”。企业常见卡点是：API 能调，但跨系统任务仍靠人工串联。

原因是：Agent 需要规划对象是**能力节点**（能完成什么业务结果），而不是零散端点。

**判断标准（可自查）：**

- 能否把关键任务抽成 10~20 个高复用能力节点？
- 每个能力节点是否声明：目标语义、前置条件、约束、失败语义、审计信息？
- 能否形成能力之间的依赖/替代/约束关系图？

---

## 4. 底座四：Observable（决策与执行要可追踪、可复盘、可运营）

企业不怕自动化“会失败”，怕的是：失败了**不知道为什么**，也无法追责与改进。

Agent 系统的可观测性至少要能回答：

- 为什么走这条路径？
- 用了哪些上下文、调用了哪些能力？
- 失败/转人工发生在哪一步？
- 这次执行成本多少、成功率如何？

---

## 一句话总结

LLM 决定智能上限，但 **API / Schema / Capability / Observable** 决定系统能不能进生产。
