# 03｜API Catalog vs Capability Catalog：接口很多但业务仍然做不完

> 适合发布：架构/平台圈层（收藏率高）

很多企业“已经 API 化”但 Agent 仍然很难落地，根因常常是：

- API 回答的是：**怎么调**
- Capability 回答的是：**能完成什么业务结果**

当系统只停留在 API Catalog（接口目录），Agent 看到的是一堆端点；它必须自己猜：先调哪个、失败怎么补偿、哪些动作需要审批。

当系统升级为 Capability Catalog（能力目录），Agent 看到的是一组“可规划、可组合、可治理”的业务能力节点。

---

## 1) 为什么“接口很多”仍然“业务做不完”

典型症状：

- 能查到数据，但难以推进状态（缺少关键写操作能力/权限）
- 能调用单点系统，但跨系统闭环仍靠人工串联
- 调用成功率不低，但端到端任务成功率很低（语义/约束缺失）

---

## 2) 能力节点最小结构（建议写进能力定义）

一个可被 Agent 稳定消费的能力节点至少要声明：

- 目标语义（要完成什么业务结果）
- 输入/输出契约（schema）
- 前置条件（状态/权限/业务规则）
- 执行约束（同步/异步、幂等、超时、重试）
- 失败语义（可恢复/不可恢复）
- 治理信息（审计字段、风险等级、责任主体）

---

## 3) 从“接口清单”到“能力目录”的迁移路径

建议不要一上来做大而全建模：

1. 选 1 条高价值链路（异常闭环/缺陷治理/审批流转）
2. 抽取 10~20 个高复用能力节点
3. 先把 schema + 权限 + 审计绑到能力层
4. 再逐步形成能力图（依赖/替代/约束关系）
