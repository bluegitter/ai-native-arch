# 20｜多智能体落地的关键：从“对话协作”转向“工件协作”（ChatDev / MetaGPT 的启发）

很多多智能体项目一开始很惊艳：几个角色聊着聊着就把需求、设计、代码都“生成”出来了。

但真正落地时最容易崩在两点：

- **对话不可复用**：你很难把“聊天记录”当作可靠交付物
- **质量不可控**：没有测试、没有版本、没有验收标准，只有看起来合理的文本

因此更稳的工程路线是：把协作从“对话中心”转为“工件中心”。

---

## 1) 代表性论文：ChatDev 与 MetaGPT 分别在“产品流程”与“工件结构”上给了答案

### A. ChatDev：用沟通链条（chat chain）组织软件开发团队

- **ChatDev: Communicative Agents for Software Development**
  - arXiv: https://arxiv.org/abs/2307.07924

工程启发：

- 它强调“该沟通什么、怎么沟通”，让团队对话更像真实研发流程
- 但真正上线时，你还需要把“对话产物”落到可验证工件：PRD、设计文档、代码、测试报告

### B. MetaGPT：把人类软件工程角色与工件显式化

- **MetaGPT: Meta Programming for A Multi-Agent Collaborative Framework**
  - arXiv: https://arxiv.org/abs/2308.00352

工程启发：

- 重要的不只是角色（PM/架构师/工程师/QA），更是：每个角色输出的**工件模板**
- 工件模板一旦结构化，就能：校验、diff、回滚、复用、做质量门禁

---

## 2) 工件中心协作的三要素：模板、版本、验收

把多智能体从“能聊”升级到“能交付”，最关键是三件事：

1. **模板化（Schema）**
   - PRD/设计/任务分解/测试用例/变更说明，都要有结构化字段

2. **版本化（Versioning）**
   - 工件必须可 diff、可回滚（Git 或等价机制）

3. **验收化（Acceptance）**
   - 每个工件要有验收标准：测试通过、静态检查通过、接口返回成功等

当这三件事存在，多智能体的对话才只是“生产流水线的一部分”。

---

## 3) 映射到本仓库五层架构：多智能体的软件工程 = Capability + Runtime 的组合拳

- **Interface 层**：代码仓库、CI、issue、知识库等系统要可调用
- **Capability 层**：每个角色=一组能力（写 PRD、画架构、写代码、跑测试、写报告）
- **Data & Memory**：共享的是“工件”，不是无限聊天上下文
- **Runtime**：流程编排（阶段门禁 + 失败回退 + 人工介入）
- **Governance**：权限（谁能 merge、谁能发布）、审计（谁做了什么）

---

## 4) 一个很实用的落地建议：让“QA/Reviewer”先上线

如果你要在企业里试点多智能体，最稳的切入不是“自动写全栈系统”，而是：

- 先让多智能体做 **评审与复核**：代码 review、PRD 评审、测试用例补全、风险检查
- 这些任务天然适合分工、并且“失败成本”低，容易纳入治理

等你把工件流与门禁机制跑顺，再把它扩展到自动生成与自动执行。
