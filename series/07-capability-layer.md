# 07｜Capability 层：能力目录、能力图与可治理编排

如果说 Interface 层解决的是“能不能连上”，那么 Capability 层解决的是：**连上之后能不能正确做完任务**。

很多 Agent 项目接口打通后仍然难以规模化，原因通常不是模型，而是缺少统一能力抽象。

---

## 1) 从 API Catalog 到 Capability Catalog

- API Catalog：有哪些端点
- Capability Catalog：哪些业务目标可被完成

Capability 的价值在于把：

- 前置条件
- 约束规则
- 失败语义
- 审计责任

这些“隐性经验”，前置成结构化资产。

---

## 2) Capability Registry：从文档中心到控制平面

生产级 Registry 需要承担：

- 生命周期：发布/灰度/下线
- 版本治理：兼容策略、破坏性变更检测
- 策略绑定：权限/风控/审计规则绑定到能力层
- 发现与路由：让 Agent 能按条件筛选能力

---

## 3) Capability Graph：让复杂任务变成“可规划”的组合问题

复杂任务不是“串几个 API”，而是：

- 条件分支
- 失败切换
- 替代路径
- 人工确认

用图表达三类关系会非常关键：

- 依赖（必须先做）
- 替代（可换实现）
- 约束（满足条件才可走）

---

## 4) 建设路径：先做小而可用

建议做法：

- 选 1 条高价值链路
- 抽 10–20 个高复用节点
- 先上 Registry + schema + 策略绑定
- 再逐步沉淀图式关系
