# 13｜读 Every《Agent-native Architectures》：从“摩天楼软件”到“花园软件”——以及它对企业 Agent-Native 的启发

> 参考文章：Every / Chain of Thought《Agent-native Architectures: How to Build Apps After the End of Code》
> https://every.to/chain-of-thought/agent-native-architectures-how-to-build-apps-after-the-end-of-code

这篇文章最大的价值不在“技术细节”，而在于它提供了一个非常好用的心智模型，帮助你向团队解释：

- 为什么 Agent-Native 不是“加一个 AI 功能”
- 为什么未来的软件会更像“可塑的生命体”，而不是“钢筋水泥的建筑”
- 为什么“写代码”在某些场景会从中心退到边缘

下面我把它的核心观点抽成可复述、可落地的版本，并补上企业语境下的边界条件。

---

## 1) 一个抓人的比喻：传统软件像摩天楼，Agent-Native 像花园

Every 的类比是：

- **传统软件（skyscraper）**：先画蓝图 → 打地基 → 每根梁都要承重计算 → 一切按规格建造。
- **Agent-Native 软件（garden）**：更像种花园——你准备土壤、阳光、水；你修剪、除草；但系统会“生长”为你无法完全预先规定的形态。

这个比喻很适合用来解释“为什么很多工程师天然不适应 Agent-Native”——因为我们过去二十年擅长的是把不确定性压扁成确定性。

---

## 2) 核心命题：Agent-Native 的核心不是代码，而是 Agent

文章里一句很关键：

> The core of an agent-native architecture is not code. Instead, the core is an agent.

在这个视角下：

- App 的“功能”不再主要是写死的步骤和 UI 路径
- 而是**给 Agent 的提示（prompts）/目标描述**：告诉它“要达成什么结果”，而不是“必须按哪几步做”

这也解释了为什么它说“开发者只需要命名 what，Agent 负责 how”。

---

## 3) “可塑性”是第一特性：用语言改变软件行为

Every 强调 agent-native apps 的一个优势是 **malleable（可塑）**：

- 用户/开发者可以通过改变文字（目标、约束、策略）来改变系统行为
- 软件从“只能由少数人开发”变成“更多人可共同塑造”

在企业语境里，把这句话翻译成工程要求就是：

- 你需要把“规则与策略”从代码里抽出来，让它**可配置、可审计、可演进**
- 否则所谓“可塑性”只会变成“不可控”

---

## 4) Every 的观点与企业落地的差异：花园可以自由生长，但企业必须可控

Every 这篇文章偏产品哲学与工程实践结合，但它也隐含了一个企业会追问的问题：

> “How can we allow software so much freedom?”

在企业里，这个问题必须被工程化回答：

- 哪些动作允许“花园式自由”（动态规划）
- 哪些动作必须回到“摩天楼式确定性”（固定流程/审批/强约束）

因此更现实的企业形态通常是：

- **Workflow（确定性骨架） + 自主编排 Agent（弹性层）**
- 再由 **Runtime + Governance** 把自由收进笼子里（状态机、补偿、审计、策略引擎）

---

## 5) 用我们的语言重述它：花园要长得好，需要“土壤、阳光、水、修剪”

把 Every 的比喻映射到企业 Agent-Native 的四原则，可以得到一个很好讲的框架：

- **API-first = 土壤**：没有可调用能力，再聪明也长不出结果
- **Schema-first = 园艺规则**：没有可校验语义，长出来的东西可能“看似健康但有毒”（调用成功但业务错）
- **Capability-based = 花园路径/区域规划**：把离散接口组织成可规划能力网络
- **Observable = 修剪与监测**：持续观测、复盘、修剪，花园才不会失控

这套映射非常适合你写在文章里，用来把“花园哲学”落到工程底座。

---

## 6) 一个实用的落地建议：把“特性”写成“目标提示”，把“约束”写成“可验证规则”

Every 的思路容易被误用成“把业务都交给模型自己想”。

更稳的落地方式是：

- Feature/能力用自然语言表达**结果目标**（what）
- 同时把关键约束前置为**可验证规则**（schema、策略、审批、审计字段）

这样你既能获得可塑性（更快迭代），又不至于牺牲可控性。

在你的业务里：

- 哪些部分必须保持“摩天楼式确定性”（强流程/强审计）？
